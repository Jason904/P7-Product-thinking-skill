# 4. Fact Confidence Level

## 4.1 为什么需要事实可信度等级

Hermes 每天会使用 AI HOT、搜索 API、GitHub、官方博客、媒体、社区讨论等信息源。

但不同来源的可信度不同。

必须避免：

```text
把 AI HOT 摘要当事实
把博主解读当事实
把社区传闻当事实
把未核验产品爆料当事实
```

因此每条关键事实必须标注事实可信度等级。

------

## 4.2 事实可信度等级

| 等级 | 来源类型                                                     | 是否可用于最终判断     | 是否需要继续核验                     |
| ---- | ------------------------------------------------------------ | ---------------------- | ------------------------------------ |
| A 级 | 官方公告 / 官方文档 / GitHub repo / 论文原文 / Release Notes / 产品官网 | 是                     | 通常不需要，但重要结论仍建议交叉核验 |
| B 级 | 权威媒体 / 创始人公开发言 / 官方 Demo / 公司高管公开采访     | 可以，但最好交叉验证   | 视情况需要                           |
| C 级 | 社区讨论 / AI HOT 摘要 / 博主解读 / 二手转述 / 非官方测评    | 不建议单独用于最终判断 | 是                                   |
| D 级 | 未核验传闻 / 截图爆料 / 无来源说法 / 群聊转述                | 否                     | 是，且核验前不得支撑判断             |

------

## 4.3 输出时必须标注

对于每个深度 case 的背景事实，必须输出：

```text
事实等级：A / B / C / D
是否可用于最终判断：是 / 否
是否需要继续核验：是 / 否
```

------

## 4.4 事实使用规则

### A 级事实

# 6. Case Selection Score

## 6.1 为什么需要 Case Selection Score

Hermes 每天会从 AI HOT、搜索 API、GitHub、产品官网、官方博客、Release Notes、新闻、社区讨论中检索大量信息。

但最终每天只做 3 个深度 case。

因此必须明确回答：

```text
为什么选这 3 个？
为什么不是别的？
这个 case 对用户的 P7+ 能力训练有什么价值？
```

Hermes 的目标不是选择“最热 case”，而是选择：

> 最能训练用户判断力、系统思维、趋势推演、机会判断和资产沉淀能力的 case。

------

## 6.2 Case Selection Score 打分表

每个候选 case 用 5 个维度打分，每项 1-5 分，总分 25 分。

| 维度       | 核心问题                               | 分值 |
| ---------- | -------------------------------------- | ---- |
| 相关性     | 是否贴近用户长期关注方向？             | 1-5  |
| 信号强度   | 是否代表行业真实变化，而不是短期噪音？ | 1-5  |
| 训练价值   | 是否适合训练 P7+ 判断力？              | 1-5  |
| 可验证性   | 是否有足够事实来源支撑分析？           | 1-5  |
| 资产化价值 | 是否能沉淀为面试 / 项目 / 方法论资产？ | 1-5  |

------

## 6.3 评分解释

### 相关性

判断这个 case 是否贴近用户长期方向：

- AI 产品化闭环
- Agent / Workflow / Skill / MCP
- AI Coding / 工程治理
- Harness / Eval / Gate
- 3D AI / 3D Agent / 空间智能
- 心理测评 / AI 心理效果评价 / 心理评估
- GitHub / 开源趋势
- AI 商业化 / 增长 / 成本结构
- 教育、心理、内容、企业软件、机器人、具身智能
- Pose Director / 多模态 3D / 自然语言姿态生成

评分标准：

```text
1 分：与用户长期方向弱相关
3 分：有一定关联，但不是核心方向
5 分：高度贴合用户长期能力建设主线
```

------

### 信号强度

判断这个 case 是否代表真实变化。

看：

- 是否是产品正式发布
- 是否是官方战略动作
- 是否有明显 GitHub 增长
- 是否有多方讨论
- 是否改变成本结构、工作流、商业模式或用户行为
- 是否代表技术范式或产业链变化

评分标准：

```text
1 分：只是单点噪音或普通更新
3 分：有一定信号，但趋势尚不明确
5 分：明显代表行业、产品、技术或商业变化
```

### 训练价值

看它是否能训练：

- 问题重定义

- 价值流向
- 做 / 不做 / 先验证
- 面试表达
- 取舍判断

评分标准：

```text
1 分：只能做信息了解
3 分：能做产品分析
5 分：非常适合训练 P7+ 判断力
```

------

### 可验证性

判断这个 case 是否有足够事实来源。

看：

- 是否有官方公告
- 是否有 GitHub repo
- 是否有 Release Notes
- 是否有论文原文
- 是否有产品官网
- 是否有权威媒体
- 是否只能看到社区讨论或二手摘要

评分标准：

```text
1 分：主要来自未核验传闻
3 分：有二手信息，但需要补充核验
5 分：有高可信事实来源支撑
```

------

### 资产化价值

判断这个 case 是否能沉淀为长期资产。

看它是否可以转化为：

- 面试回答

评分标准：

```text
1 分：看完即过，不容易沉淀

```

# 12. 6 层结构化总结

8 问是怎么想，6 层是怎么讲。

每次都要把前面的推理整理成 6 层。

## 1. 现象

回答：

```text
发生了什么？
```

要求：

- 只讲事实和可观察变化
- 不夹杂过多判断
- 明确对象、场景、行为变化

输出：

```text
【现象】
我观察到：
```

------

## 2. 原因

回答：

```text
为什么发生？
```

要求：

- 不是单点归因
- 至少拆用户、技术、商业、组织、风险等维度
- 找出核心驱动因素

输出：

```text
【原因】
它不是由单一因素导致，而是：

其中最核心的驱动是：
```

------

## 3. 本质

回答：

```text
真正矛盾是什么？
```

要求：

- 必须用“这不是 X，而是 Y”表达
- 把表层问题抽象成底层价值或底层矛盾

输出：

```text
【本质】
表面上是：
本质上是：

一句话本质判断：
```

------

## 4. 系统

回答：

```text
哪些力量共同作用？
```

要求：

- 必须讲推力、阻力、瓶颈、放大器
- 必须讲系统关系，而不是只罗列因素
- 最好写出一条关系链

输出：

```text
【系统】
关键参与因素包括：

核心系统关系是：

推力：
阻力：
瓶颈：
放大器：
```

------

## 5. 趋势

回答：

```text
这个系统未来会如何演化？
```

要求：

- 从系统变量推导，不要拍脑袋
- 明确短期、中期、长期路径

输出：

```text
【趋势】
我判断它会从：

现在 → 阶段 1 → 阶段 2 → 长期形态

长期趋势是：
```

------

## 6. 机会

回答：

```text
最大价值缺口在哪里？
```

要求：

- 不能只说做功能
- 必须指出价值控制点
- 必须说明为什么这个机会更大

输出：

```text
【机会】
最大机会不在：
而在：

因为：
```

```text
价值从单次内容生成，流向企业知识库、品牌治理、协作审核和内容生产工作流。
```

### 做 / 不做 / 先验证

必须有明确取舍。

示例：

```text
做：企业知识库 + 模板 + 审核流。
不做：只做通用 PPT 生成器。
先验证：企业是否愿意为品牌一致性和事实可控付费。
```

# 24. 最终记忆口令

每次分析前，先默念：

```text
先问谁，再问场景；
先看成本，再看收益；
先找矛盾，再看系统；
先推演化，再找机会；
先做取舍，最后方案。
```

每次表达时，用：

```text
现象 → 原因 → 本质 → 系统 → 趋势 → 机会
```

每次判断时，用：

```text
这不是 X 问题，而是 Y 问题。
最大机会不在 A，而在 B。
所以不应该优先做 C，而应该先验证 D。
```

GitHub 趋势判断时，用：

```text
这不是一个 repo 火了，
而是开发者在寻找某种新工作流 / 新基础设施 / 新控制点。
```

3D AI 判断时，用：

```text
这不是一次性生成 3D 资产，
而是把空间意图转化为可编辑、可复用、可交付的 3D 工作流。
```

AI 心理判断时，用：

```text
这不是 AI 陪聊，
而是安全、可验证、可分级、可转介的心理支持与评估系统。
```

Case 选择判断时，用：

```text
我真正需要的不是最热 case，
而是最能训练我判断力、并能沉淀为长期资产的 case。
```

# 27. Skill v5.0 最终目标

Hermes v5.0 的最终目标不是帮用户每天看更多信息，而是帮用户长期建立：

```text
AI PM 判断力案例库
```

这个案例库由每天的深度 case 组成，每个 case 都必须经过：

```text
信息源发现
→ Case Selection Score
→ Fact Confidence Level
→ 3 个深度 case 选择
→ 8 问显性推理
→ 模型调用
→ 6 层表达
→ 判断取舍
→ Case Asset Card
→ 遗忘曲线复现
→ Notion / Obsidian / 本地知识库沉淀
```

最终用户要形成的能力是：

```text
看到一个热点，不只是知道它火；
看到一个产品，不只是知道它做了什么；
看到一个 GitHub 项目，不只是知道它 star 涨了；
看到一个心理 AI，不只是觉得体验好；
看到一个 3D AI，不只是看生成效果。

而是能判断：
谁的问题？
什么场景？
损失什么？
想得到什么？
为什么卡住？
哪些系统因素共同作用？
未来怎么演化？
价值会流向哪里？
应该做什么？
不做什么？
先验证什么？
如何沉淀为自己的项目、面试和方法论资产？
```
